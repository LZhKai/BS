\cabstract{

   随着城市化进程的不断加快，居民小区机动车数量迅速增长，传统依赖人工登记和人工巡查的车辆管理方式已难以满足智慧社区对高效率、高准确性和高安全性的管理需求。针对小区车辆出入管理效率低、实时性不足以及智能化水平有限等问题，本文设计并实现了一种基于计算机视觉与流式计算的智能小区车辆管理系统。系统以 Spring Boot 为核心后端框架，结合 Vue 前端技术构建前后端分离架构，利用 MySQL 实现车辆与用户信息的统一管理，并通过 Spark Streaming 对车辆相关数据流进行实时处理。在功能层面，系统集成了车辆识别、视频流监控、车辆信息管理以及基于 RAG 的智能问答模块，实现了车辆管理的自动化、实时化与智能化。实验与系统测试结果表明，该系统在车辆识别准确率、数据处理实时性和系统稳定性方面均具有良好表现，可为智慧社区车辆管理提供一种可行的技术方案。

}
\ckeywords{计算机视觉；车辆识别；流式计算；Spark Streaming；RAG}

\eabstract{
   With the rapid acceleration of urbanization, the number of motor vehicles in residential communities has increased significantly. Traditional vehicle management methods that rely on manual registration and inspection can no longer meet the requirements of intelligent communities in terms of efficiency, accuracy, and real-time performance. To address the problems of low management efficiency, insufficient real-time processing, and limited intelligence in residential vehicle management, this paper designs and implements an intelligent residential vehicle management system based on computer vision and streaming computing.
   
   The system adopts Spring Boot as the core backend framework and Vue as the frontend framework to build a front-end and back-end separated architecture. MySQL is used for unified management of vehicle and user information, while Spark Streaming is introduced to process vehicle-related data streams in real time. In terms of functionality, the system integrates license plate recognition, video stream monitoring, vehicle information management, and an intelligent question-answering module based on Retrieval-Augmented Generation (RAG), realizing automation, real-time processing, and intelligence in vehicle management.
   
   Experimental results and system testing show that the proposed system performs well in license plate recognition accuracy, real-time data processing capability, and overall system stability, providing a feasible technical solution for intelligent vehicle management in smart residential communities.

}

\chapter{引言}

\section{课题的背景与意义}

随着城市化进程的不断推进，居民小区机动车保有量持续增长，车辆出入管理、停车资源配置以及安全监管等问题日益突出。传统依赖人工登记与人工巡查的管理方式效率低、信息更新滞后，统计误差较大，难以满足智慧社区对精细化和信息化管理的需求。

在智慧城市与智慧社区建设背景下，智能小区车辆管理系统逐渐成为提升社区治理能力的重要技术手段。该类系统通常依托计算机视觉技术实现车牌自动识别与车辆状态感知，并结合数据库技术完成车辆信息的集中存储与统一管理，从而降低人工成本并提升管理效率。在智能停车与车辆管理系统研究中，车辆检测、车牌识别与停车引导的协同处理方法能够显著提升复杂环境下系统的识别准确率和整体运行效率\overcite{Gupta2025Parking}。多目标车辆识别研究表明，基于深度学习的分类与识别方法能够在不同车辆类型和多目标场景下实现较高识别精度\overcite{Wu2024VehicleDL}。

在系统实现层面，Spring Boot 框架凭借模块化设计和可扩展性，为高并发管理系统提供了实践经验，其架构支持模块独立开发和部署，并提供统一配置和安全策略，有助于提升系统的可维护性和扩展能力\overcite{Miguel2024SpringBoot}。结合 Spark Streaming 的流式计算能力，可处理大规模实时数据流，并保证计算一致性与容错能力，从而提升整体系统的数据处理效率和响应性能\overcite{Daksa2025Streaming}。

此外，大语言模型与检索增强生成（RAG）技术的引入，为智能问答和数据检索提供了新的研究方向，可辅助管理人员进行高效决策支持\overcite{Feng2025RAG,Wang2025LLMJavaWeb}。

\section{国内外研究现状和分析}

在智慧社区和智能交通背景下，车辆信息管理、车牌自动识别、视频监控分析及实时数据处理等技术是研究重点。国内外已有大量研究成果，为智能小区车辆管理系统的设计与实现提供了理论基础和技术支撑，但在系统集成度、算法鲁棒性及实际应用适配方面仍存在优化空间。

\subsection{国外研究现状}

在智能停车与车辆管理系统研究中，车辆检测、车牌识别与停车引导的协同处理方法被证明可以有效提升复杂环境下系统的识别准确率与整体运行效率\overcite{Gupta2025Parking}。该研究通过将车辆检测和车牌识别方法与停车引导策略结合，实现了系统在多阶段处理流程中的高效协同，提高了数据处理的一致性和系统响应速度。多目标车辆识别问题的研究表明，基于深度学习的分类与识别方法能够适应不同车型、不同角度，从而在复杂交通环境下实现稳定、可靠的车辆识别\overcite{Wu2024VehicleDL}。这些方法利用卷积神经网络进行特征提取和多类别分类，显著增强了系统在多车辆和不同场景下的适用性。

在系统架构与工程实现方面，Spring Boot 在企业级 Web 应用中的模块化设计、可扩展性及安全机制为高并发管理系统提供了实践经验\overcite{Miguel2024SpringBoot}。其模块化结构使得不同功能模块可以独立开发和部署，同时便于系统升级和维护。在实时数据处理领域，对 Apache Spark Streaming 与 Flink 的性能对比分析显示，Spark Streaming 在高吞吐量、大规模数据流处理和容错能力方面具有明显优势\overcite{Daksa2025Streaming,Fang2023SparkKafka}。结合 Spark Streaming 的流处理能力，系统能够在数据流量剧增时保持稳定运行，为车辆出入管理和停车数据的实时处理提供技术保障。将深度学习算法与实时数据处理框架结合，可以在保证高识别率的前提下，实现车辆检测和数据处理的高效协同，从而提升智能交通和停车管理系统的整体性能\overcite{Gupta2025Parking}。

\subsection{国内研究现状}

国内在 Web 管理系统开发、车辆识别算法以及流式计算应用等方面开展了大量研究。在前端与系统架构方面，Vue.js 框架的组件化开发模式被广泛应用，其模块化设计和组件复用特性，使得界面开发更加系统化、结构清晰，同时支持可维护的代码组织和模块独立升级\overcite{Xia2025Vue,Duan2024JavaWeb}。在实际工程应用中，Spring Boot 与 Vue 技术组合在事故报告数据库系统的开发中被验证为可行方案，该组合在数据处理、模块化开发和系统架构优化方面提供了有效支撑，并且在系统开发实践中体现了良好的工程适用性\overcite{Xue2025AccidentDB,Shan2021DBCourse}。从空间信息系统的角度，Spring Boot 框架在 GIS 可视化系统中的应用经验表明，其模块化架构能够支撑数据管理、数据交互和可视化功能的整合，为 Web 系统在处理复杂数据时提供了工程参考\overcite{Chen2024GIS}。此外，大语言模型在 Java Web 项目中的辅助分析和系统环境安装能力，展示了模型在系统开发中的应用潜力，为研究系统开发效率和代码生成方法提供了新的技术参考\overcite{Wang2025LLMJavaWeb}。

在数据管理与信息处理方面，MySQL 数据库的流式备份技术能够保障数据安全与稳定性，并支持数据恢复和容错管理\overcite{Yu2025MySQL}。同时，OCR 技术在信息自动识别和核验方面的应用，展示了文档处理与信息提取的自动化可能性\overcite{Huang2025OCR}。在车辆识别与视频处理领域，深度学习算法在复杂场景下实现了高精度车辆检测，并通过轻量化神经网络在保证识别精度的同时降低了计算资源消耗\overcite{Gao2025VehicleDL,Song2025LightNet}。视频采集与传输系统的工程实现提供了可行的技术方案，并在数据流获取、传输和处理方面展示了系统的技术实现方法\overcite{Yang2025Video}。结合 Spark Streaming 与物联网平台的数据处理实践，可以在处理连续数据流和大规模数据时体现出系统的实时性与可扩展性\overcite{Huang2023ThingsBoard,Ma2021Trajectory}。

在系统整体设计方面，基于 Spring Boot 的车辆管理系统在多模块协作、权限管理以及数据处理方面提供了工程实践参考，展现了框架在系统集成和模块化开发中的应用价值\overcite{Du2022Parking}。此外，大语言模型与 RAG 知识增强问答系统在低资源场景下展示了其问答能力，为研究智能化信息获取与系统辅助分析提供了技术参考\overcite{Chen2025ChatGLM,Feng2025RAG}。

\section{本课题的主要研究内容}

围绕智能小区车辆管理的实际需求，本文以计算机视觉与流式计算技术为核心，开展以下研究工作：（1）系统总体架构设计。采用前后端分离架构，前端基于 Vue 实现数据展示与交互，后端基于 Spring Boot 实现传统业务逻辑处理与接口服务，Python 后端提供视觉识别与 RAG 服务，并结合 MySQL 数据库存储系统核心数据，保证系统的可扩展性和可维护性。（2）车辆信息管理与检测。通过计算机视觉技术实现车辆目标检测与车牌识别，将车牌识别结果经人工确认后同步至车辆档案，并与车主管理、停车记录管理模块配合，提升车辆出入信息的登记效率与可追溯性。（3）车辆监控与实时聚合。引入 Spark Structured Streaming 对监控检测事件进行滑动窗口聚合，通过 Socket.IO 将聚合指标实时推送前端，提高系统在高并发场景下的实时性和稳定性。（4）RAG 智能问答模块设计。结合向量数据库与大语言模型，实现基于 RAG 的智能问答和历史追踪能力，为管理人员提供便捷的信息查询与辅助决策支持。

\section{系统视觉感知基础}

车辆检测是系统感知层入口，目标是在复杂场景下稳定输出目标框、类别与置信度信息。与静态图像不同，视频场景更强调时序连续性，单帧结果容易受到光照变化、遮挡、反光和运动模糊影响。为提升工程可用性，系统在不改动模型结构的前提下，通过抽帧推理、多目标跟踪与类别归一化三层机制将单帧感知结果转化为稳定的跟踪序列，避免将模型噪声直接传递到统计层。

本系统采用 YOLOv8n 作为车辆检测模型。YOLOv8 属于单阶段目标检测算法，采用 anchor-free 设计与 CSPDarknet 骨干网络进行多尺度特征提取，在检测精度与推理速度之间取得较好平衡。相较于 YOLOv5 等前代模型，YOLOv8 在小目标检测与密集场景下的召回率有显著提升。选用 YOLOv8n（nano 变体）是基于轻量化考虑：该变体参数量较小、推理延迟低，适合在单 GPU 甚至 CPU 环境下保持实时帧率，满足小区监控场景对连续处理的需求。在工程部署中，视频处理模块对视频源采取固定间隔的抽帧推理策略，中间帧沿用上一次检测结果并重绘检测框，在降低算力开销的同时保持画面流畅度。

\subsection{多目标跟踪与稳定计数约束}

可用于业务统计的检测链路应满足语义清晰、时序连续和可追踪三项要求。若目标 ID 频繁跳变或类别映射不稳定，将直接导致事件抽取层判断失真。为此系统在视频处理模块中实现了基于 IoU 与中心点距离的轻量级多目标跟踪器：对每一帧的新检测框，先在已有轨迹集合中匹配最近的历史轨迹并继承其 \texttt{track\_id}，未匹配的检测框则分配新编号；轨迹通过 TTL 机制处理短暂丢失，超过生命周期的轨迹被回收。为抑制闪现型误检，系统引入最小命中次数约束，只有当同一轨迹持续出现若干帧后才将其计入唯一车辆统计。该约束使计数依据具备可复核性，避免了将单帧抖动直接放大为计数误差。

\subsection{复杂场景鲁棒性处理}

在夜间、逆光和拥堵等复杂场景下，单帧检测结果容易受到光照变化、运动模糊与目标相互遮挡影响，出现漏检、误检或类别抖动现象。系统在跟踪层通过置信度阈值过滤与最小命中次数约束抑制闪现型误检，并借助生命周期机制（TTL）允许轨迹在短时遮挡后重新匹配回原编号，避免同一目标被拆分为多条新轨迹而重复累加。类别上则对检测输出执行归一化映射，将轿车、货车、公交车和摩托车等类别收敛到统一枚举集合，防止类别字符串差异导致统计口径漂移。上述机制共同保证在保持实时性的前提下，跟踪序列具备可复核的时序稳定性。

\section{系统统计语义与事件抽取机制}

本文将车流量定义为"单位窗口内有效通行事件数"，而非检测框数量。检测输出是感知结果，统计输出是业务指标，两者必须通过事件层转换。该定义是解决口径不一致问题的关键。

\subsection{事件抽取规则}

事件抽取以跟踪轨迹为最小单位进行去重：只有当某条轨迹累计命中次数达到最小阈值且首次被判定为"稳定轨迹"时，系统才将其登记为一次有效车辆事件，后续同一 \texttt{track\_id} 的再次观测仅用于刷新最近位置、置信度与类别分布，不再重复计数。对短时遮挡后重现的目标，跟踪器通过 IoU 与中心点距离匹配保留其原 \texttt{track\_id}，从而避免同一车辆被拆分为多条轨迹重复累加。这一规则使检测层输出的"每帧车辆数"与业务层关注的"区间内唯一车辆数"之间具备清晰的转换关系，便于与后续窗口聚合结果相互对账。

\subsection{帧级明细与窗口级聚合的双粒度口径}

系统在输出侧采用两种时间粒度并行的方式。一种是帧级明细：每识别完一帧画面，系统立刻把当前这帧的检测结果（包括车辆框、车型、实时累计的唯一车辆数）推送到监控画面上，让值班人员能第一时间看到画面叠加效果。另一种是窗口级聚合：系统按固定的时间间隔对最近一段时间内的所有识别事件做整体统计，输出总数、车型分布、平均置信度等指标，用于刷新统计看板的卡片和趋势图。两种粒度都来自同一份识别事件数据，只是输出的时机和统计口径不同，因此画面看到的实时情况和看板上展示的汇总指标能够相互印证。当聚合链路因资源波动短时中断时，前端还可以回退到帧级明细继续刷新图表，保证看板不出现长时间空白。

\section{系统业务服务基础（Spring Boot）}

Java 后端采用 Spring Boot 3 框架构建，利用其自动配置（Auto-Configuration）机制简化项目初始化与依赖管理。Spring Boot 通过 \texttt{@SpringBootApplication} 注解整合组件扫描、自动配置与配置属性绑定，使开发者无需编写大量 XML 配置即可快速搭建 RESTful 服务。

在数据访问层，系统采用 MyBatis-Plus 作为 ORM 框架。MyBatis-Plus 在 MyBatis 基础上提供了通用 CRUD 接口（\texttt{BaseMapper}）、Lambda 条件构造器（\texttt{LambdaQueryWrapper}）和内置分页插件，减少了手写 SQL 的工作量。实体类通过 \texttt{@TableName}、\texttt{@TableId} 等注解与数据库表建立映射关系，分页查询通过 \texttt{Page<T>} 对象封装页码与每页数量，查询结果由框架自动填充总条数与记录列表。

在接口设计层面，系统遵循 RESTful 规范：资源以名词路径标识（如 \nolinkurl{/api/vehicle}），操作通过 HTTP 方法区分（GET 查询、POST 新增、PUT 更新、DELETE 删除）。接口统一返回 \texttt{Result<T>} 包装结构，包含状态码、提示信息与业务数据三个字段，便于前端统一拦截和错误处理。参数校验采用 Bean Validation 注解（如 \texttt{@NotBlank}、\texttt{@NotNull}），在 Controller 层即完成输入合法性检查。

\section{系统视觉与流式服务基础（Python）}

Python 服务基于 Flask 微框架构建，采用蓝图（Blueprint）机制将视频处理、车牌识别与 RAG 问答三个功能域拆分为独立模块，各模块在 \texttt{app.py} 中统一注册并挂载到 \texttt{/api} 前缀下。蓝图机制使各模块具备独立的路由命名空间和视图函数，便于按需扩展与维护。

实时通信层采用 Flask-SocketIO，在 HTTP 服务基础上提供全双工的 WebSocket 通道。系统注册了 \texttt{/stream/monitor}（检测明细与聚合统计）和 \texttt{/stream/monitor-video}（视频帧流）两个命名空间，前端通过 \texttt{socket.io-client} 连接后即可接收服务端主动推送的数据，无需轮询。该机制满足实时监控场景对低延迟、高频率数据更新的需求。

在流式计算层面，系统以视频流检测结果作为数据源，引入 Spark Structured Streaming 实现实时事件聚合。完整的数据流管道如下：视频处理模块（\texttt{VideoProcessor}）从监控视频源逐帧读取画面，每隔固定帧间隔将当前帧送入 YOLOv8 目标检测模型获取车辆边界框与类别信息，随后经多目标跟踪器（基于 IoU 与中心点距离匹配）为每个目标分配稳定的轨迹编号（\texttt{track\_id}）。跟踪完成后，每个检测到的车辆被转化为一条结构化事件，事件字段包含时间戳（\texttt{ts}）、轨迹编号（\texttt{track\_id}）、归一化车型（\texttt{vehicle\_type}）、置信度（\texttt{confidence}）和帧编号（\texttt{frame\_id}），通过 \texttt{append\_event()} 写入 \texttt{MonitorSparkStreamer} 的内存事件队列，由此形成从视频帧到结构化事件的连续数据流。

Structured Streaming 是 Spark 提供的基于 DataFrame/Dataset API 的流处理引擎，以"微批"（micro-batch）或"连续处理"模式将无界数据流视为不断追加的关系表进行增量计算。本系统采用 JSON 文件源（File Source）作为 Spark 的数据接入方式：后台刷写线程（flush thread）定时从内存队列中取出事件，以换行分隔 JSON（NDJSON）格式通过原子写入（\texttt{os.replace}）落盘至暂存目录，Spark 通过 \nolinkurl{readStream.format("json")} 持续监听该目录并自动发现新文件，将其解析为 DataFrame。流处理管道使用 \nolinkurl{writeStream.foreachBatch()} 注册批处理回调，每个微批触发时在回调中维护一个基于时间戳的滑动窗口（默认宽度 30 秒），对窗口内事件执行 \texttt{groupBy} 聚合计算（总计数、唯一轨迹数、平均置信度、车型分布等），聚合结果通过 Socket.IO 推送至前端监控看板。触发间隔通过 \nolinkurl{trigger(processingTime)} 配置，默认为 5 秒。该设计在使用 Spark 原生流处理 API 的同时，以本地文件系统替代 Kafka 等外部消息队列，降低了小规模部署场景的基础设施复杂度。若 PySpark 环境不可用，系统自动降级为检测流即时统计，保障基本功能连续性。

\section{系统 RAG 技术基础与应用定位}

检索增强生成（Retrieval-Augmented Generation, RAG）是一种将信息检索与文本生成相结合的技术范式。其标准流程可概括为三个阶段：

（1）检索阶段（Retrieve）：将用户问题编码为向量表示，从预构建的知识库中召回语义相关的候选文档片段。本系统使用 Sentence-Transformers 模型将文本编码为固定维度的稠密向量，编码过程基于预训练的双塔 BERT 架构，能够捕获文本的深层语义信息。

（2）增强阶段（Augment）：将召回的候选片段与原始问题拼接为结构化提示词（Prompt），为生成模型提供上下文依据。提示词中包含回答约束条件（如必须基于检索结果作答、禁止编造业务数据），以控制生成质量。

（3）生成阶段（Generate）：将增强后的提示词提交至大语言模型，由模型生成自然语言回答。本系统通过 API 调用外部大模型服务，支持流式输出以降低用户感知延迟。

在向量存储层面，系统采用 Milvus 向量数据库。Milvus 支持多种近似最近邻（ANN）索引算法（如 IVF\_FLAT、HNSW），能够在大规模向量集合上实现毫秒级检索。系统在知识构建阶段将车辆主数据（车牌、品牌型号、车主信息等）编码为向量并写入 Milvus 集合，查询时通过余弦相似度召回 Top-K 候选。

RAG 模块在本系统中定位为辅助决策工具，服务于运维人员的规则查询、故障排查与文档检索需求，不直接参与车流主指标计算，因此不会影响统计口径。

\section{业务场景与核心痛点}

系统服务对象为小区物业与安保人员，业务场景包括车辆登记、停车记录管理、实时视频监控和异常处置。结合项目迭代过程，核心痛点集中在三类：（1）识别层与统计层语义不统一，导致"识别数量"和"监控统计"不一致；（2）表单输入与后端校验规则不一致，造成新增接口返回 400 错误；（3）视频任务结束缺少统一收尾，导致统计更新时机不稳定。针对上述问题，系统采用"识别—事件—聚合"三层口径，统一由后端聚合接口输出业务指标，前端只展示不重算。

\section{功能需求细化}

\subsection{车辆与车主管理}

支持车主信息维护、车辆信息维护、黑名单状态管理与快速查询。车辆与车主存在一对多关系，车辆表保存冗余的车主姓名和手机号以提升列表查询效率。

\subsection{停车记录管理}

支持进场登记、离场登记、停车时长与费用计算。新增记录时必须校验车牌与车辆 ID 关联关系，避免孤立记录写入。

\subsection{视频识别与统计}

支持实时画面目标框展示、车型显示、置信度与时间记录。统计采用"逐帧识别 + Structured Streaming 聚合"双通道：逐帧链路负责视频画框与明细展示，Structured Streaming 链路按固定触发间隔对滑动窗口内事件进行聚合并推送卡片与趋势图，兼顾实时性与稳定性。

\subsection{RAG 问答辅助}

RAG 模块采用“检索 + 生成”机制，面向车辆与车主主数据提供语义检索与问答能力，辅助管理人员快速查询信息。问答过程会记录到 \texttt{rag\_qa\_log} 便于追踪与历史展示，但不直接修改车辆、车主与停车等业务主数据，因此不会改变主业务口径。

\subsection{关键数据流}

系统数据流按业务职责划分为三条主链路：

（1）管理链路：前端表单 $\rightarrow$ Java/Python 接口 $\rightarrow$ MySQL，负责车辆、车主、停车记录等结构化业务数据的增删改查。

（2）识别链路：视频源 $\rightarrow$ \texttt{VideoProcessor} 逐帧读取 $\rightarrow$ YOLOv8 目标检测 $\rightarrow$ IoU/中心点跟踪器分配 \texttt{track\_id} $\rightarrow$ 检测明细与视频帧通过 Socket.IO 实时推送至前端。该链路以帧为粒度运行，为前端提供逐帧画面与检测框叠加展示。

（3）聚合链路（视频流 $\rightarrow$ Spark Structured Streaming）：识别链路每帧输出的检测结果在 \texttt{detection\_callback} 中被封装为结构化事件（ts/track\_id/vehicle\_type/confidence/frame\_id），通过 \texttt{append\_event()} 写入 \texttt{MonitorSparkStreamer} 内存队列 $\rightarrow$ 后台 flush 线程定时将队列事件以 NDJSON 格式原子写入暂存目录 $\rightarrow$ Spark \texttt{readStream.format("json")} 持续监听目录发现新文件 $\rightarrow$ \texttt{writeStream.foreachBatch()} 在每个微批中维护滑动时间窗口并执行聚合 $\rightarrow$ 聚合指标通过 Socket.IO 推送至前端监控看板。该链路以秒级窗口为粒度运行，将高频帧级检测事件转化为稳定的统计指标输出。

\subsection{任务状态机}

视频监控任务在实现中维护两个并列的运行标志：\texttt{VideoProcessor.is\_running} 反映视频读取与推理线程的工作状态，\texttt{MonitorSparkStreamer.\_running} 反映 Structured Streaming 查询的工作状态。两者共同构成"未启动—运行中—收尾中—已停止"四个可观察阶段：任务创建后置 \texttt{is\_running=True} 进入运行态；收到停止指令或视频读帧返回结束标志时进入收尾态，依次调用 \texttt{VideoProcessor.stop()} 终止处理线程、触发 \texttt{on\_video\_end} 回调并调用 \nolinkurl{monitor_streamer.stop()} 关闭 Streaming Query、刷写残余事件与清理临时文件；两个标志均置 \texttt{False} 后任务进入已停止态。\nolinkurl{/api/monitor/status} 接口返回的 \texttt{is\_running} 与 \texttt{spark\_enabled} 字段即是该状态机的外部视图。

\subsection{异常分支处理}

对于网络抖动、数据库瞬时不可写等场景，系统采用"重试 + 幂等"策略；对于参数错误，Java 接口返回字段级错误信息，前端据此提示用户修正输入。

\section{非功能需求量化指标与验收标准}

为避免非功能需求停留在描述层面，本文将关键非功能目标量化为可验收指标，并在测试阶段逐项核对。

在实时性方面，页面关键接口 P95 响应时间需控制在可用阈值内，确保值班人员在高峰时段仍能及时获取状态。

在一致性方面，同一任务在不同页面展示的核心统计字段必须一致，允许短时刷新延迟但不允许最终结果不一致。

在稳定性方面，连续运行任务需保持资源可回收，不出现持续性内存增长和任务僵死。

在可维护性方面，系统要求错误信息可定位到字段或模块，不接受仅返回失败而无原因提示的接口行为。前后端统一参数校验策略，接口在参数非法时返回可读错误信息，前端在表单层对必填项、时间格式和关联 ID 进行前置校验，使问题定位链路从日志猜测转变为字段级直达。

在可扩展性方面，系统要求新增视频源、扩展车型映射和新增统计维度时不影响既有接口语义。通过前后端分层和监控聚合链路解耦，项目实现了局部变更不影响全局可用的演进能力。

在验收方法上，本文采用"接口级验收 + 链路级验收 + 回归验收"三层结构。接口级验收验证单点功能正确，链路级验收验证跨模块一致，回归验收验证历史问题不复发。

\subsection{约束与索引设计}

（1）车牌唯一约束：vehicle.plate\_number，避免重复车辆档案。（2）外键约束：parking\_record.vehicle\_id 指向 vehicle.id，vehicle.owner\_id 指向 vehicle\_owner.id。（3）时间索引：parking\_record.entry\_time、plate\_recognition\_record.created\_at、rag\_qa\_log.create\_time。（4）监控聚合采用 Spark Structured Streaming 滑动窗口计算与 Socket.IO 实时推送，不依赖固定统计表落库。

\section{前端实现细节}

前端采用 Vue 3 + Element Plus 实现，按"路由层（router）—接口层（api）—页面层（views）"组织。路由在 \nolinkurl{front/src/router/index.js} 中集中声明，主功能页面包含车辆管理（\texttt{/vehicle}）、车主管理（\texttt{/owner}）、停车记录（\texttt{/parking}）、车辆监控（\texttt{/detection}）、车辆登记（\texttt{/plate-recognition}）与 RAG 问答（\texttt{/rag}）。全局布局由 \texttt{Layout.vue} 提供顶部用户信息区与左侧导航菜单，路由守卫基于 \texttt{localStorage} 的登录态进行访问控制。

\subsection{前端页面结构与导航效果}

页面整体采用"顶部栏 + 左侧菜单 + 内容区"三段式结构。侧边菜单项与路由路径一一对应，用户切换菜单后在同一主容器中切换业务页面，避免多窗口跳转造成的上下文丢失。该布局设计的工程价值在于：一是减少操作路径长度，二是让"车辆—停车—监控—RAG—登记"形成连续业务流。

\section{Java 后端实现细节}

Java 后端采用 Spring Boot 3 + MyBatis-Plus 实现 CRUD、参数校验与分页查询。项目使用 \texttt{Result<T>} 统一响应包装（含 \texttt{code/message/data}），分页结果由 \texttt{PageResult<T>} 封装（含 \texttt{total/records}）。校验逻辑集中在 Controller 层，Service 层仅负责分页条件拼装，Mapper 层直接继承 MyBatis-Plus 的 \texttt{BaseMapper}，无自定义 XML 映射。

\subsection{停车新增接口核心逻辑}

停车新增接口位于 \texttt{ParkingRecordController}，使用 \texttt{@PostMapping} 接收 \texttt{ParkingRecord} 实体。Controller 内依次校验 \texttt{vehicleId}、\texttt{plateNumber}、\texttt{entryTime} 非空，默认设置状态为 \texttt{PARKING}，并查询是否已存在同一车辆的在场记录（\texttt{status="PARKING"}），若存在则返回错误提示"该车辆已有在场停车记录，请先离场后再新增"，否则调用 \texttt{parkingRecordService.save()} 完成写入。更新接口同样检查在场记录冲突，防止同车多条在场。

\subsection{车辆新增与车主关联逻辑}

车辆新增接口位于 \texttt{VehicleController}，使用 \texttt{@Valid @RequestBody VehicleDTO} 接收输入，其中 \texttt{plateNumber} 通过 \texttt{@NotBlank} 注解进行 Bean Validation。Controller 先通过 \texttt{lambdaQuery()} 查询车牌是否已存在，再将 DTO 属性复制到 \texttt{Vehicle} 实体，若 \texttt{ownerId} 非空则从车主表查询并回填 \texttt{ownerName} 和 \texttt{ownerPhone} 冗余字段，最后调用 \texttt{vehicleService.save()} 完成写入。更新接口额外排除自身 ID 进行车牌唯一性检查，防止修改车牌为已存在值。

\section{Python 后端实现细节}

Python 负责视频推理、事件抽取、Structured Streaming 聚合触发与结束收尾。入口文件 \texttt{app.py} 注册 \texttt{video}、\texttt{plate}、\texttt{rag} 三个 Flask 蓝图，统一挂载在 \texttt{/api} 前缀下，并通过 Flask-SocketIO 注册 \texttt{/stream/monitor} 与 \texttt{/stream/monitor-video} 两个命名空间。

\subsection{视频任务启动与收尾}

视频任务由 \nolinkurl{/api/monitor/start}（兼容旧路由 \nolinkurl{/api/video/start}）启动。启动流程中先重置跟踪状态，然后尝试启动 Structured Streaming 模块（\nolinkurl{monitor_streamer.start()}），该方法初始化 SparkSession、创建 JSON 文件监听目录并启动 \nolinkurl{readStream -> writeStream.foreachBatch} 流处理管道，若 PySpark 不可用则自动降级。随后创建 \texttt{VideoProcessor} 实例并指定三个回调：\texttt{detection\_callback} 处理检测结果、\texttt{frame\_callback} 转发视频帧、\texttt{on\_video\_end} 执行收尾。接口响应中返回 \texttt{spark\_enabled} 字段指示 Spark 是否启用。视频自然结束和手动调用 \nolinkurl{/api/monitor/stop} 均会执行 \nolinkurl{monitor_streamer.stop()} 停止 Streaming Query 并完成统一收尾。

\subsection{检测回调与事件写入}

检测结果在 \texttt{video\_api.py} 的 \texttt{detection\_callback} 中处理。该函数先调用跟踪逻辑（基于 IoU 与中心点距离匹配 \texttt{track\_id}），再将每个检测到的车辆封装为结构化事件直接调用 \texttt{monitor\_streamer.append\_event()}，事件字段包括 \texttt{ts}（时间戳）、\texttt{track\_id}（轨迹编号）、\texttt{vehicle\_type}（归一化车型）、\texttt{confidence}（置信度）和 \texttt{frame\_id}（帧编号）。若 Structured Streaming 有最新聚合结果，则用其覆盖默认统计指标后通过 \texttt{send\_detection\_data()} 推送到前端。

\subsection{Structured Streaming 聚合管道实现}

本节详细说明从视频流检测事件到 Spark Structured Streaming 聚合输出的完整管道实现。该管道的核心类为 \texttt{MonitorSparkStreamer}（位于 \texttt{monitor\_streaming.py}），其内部由事件接收、文件刷写、流式读取与窗口聚合四个阶段组成。

\textbf{阶段一：事件接收与内存缓冲。}\texttt{append\_event()} 方法接收来自检测回调的结构化事件，将其追加到线程安全的内存双端队列（\texttt{deque}）中。队列设置 10000 条上限防止内存溢出，超出时丢弃最早事件。

\textbf{阶段二：后台刷写线程将事件落盘为 JSON 文件。}独立的 flush 线程以 \texttt{batch\_seconds/2} 为间隔（默认 2--3 秒）从内存队列中批量取出事件，将其序列化为换行分隔 JSON（NDJSON）格式，通过"先写临时文件、再 \texttt{os.replace()} 原子重命名"的方式写入暂存目录 \texttt{\_stream\_input/}，防止 Spark 读取到写了一半的文件。刷写线程同时负责清理已过期的旧文件（超过 2 倍窗口宽度），控制磁盘占用。

\textbf{阶段三：Spark Structured Streaming 读取文件流。}\texttt{start()} 方法初始化 SparkSession 后，通过 \nolinkurl{readStream.format("json").schema(_EVENT_SCHEMA).load(_stream_dir)} 创建流式 DataFrame。Spark 引擎持续监听暂存目录，每当发现新的 JSON 文件即自动读取并解析为包含 \texttt{ts}、\texttt{track\_id}、\texttt{vehicle\_type}、\texttt{confidence}、\texttt{frame\_id} 五个字段的 DataFrame。Schema 通过 \texttt{StructType} 预定义，避免自动推断带来的不稳定性。

\textbf{阶段四：foreachBatch 滑动窗口聚合。}流处理管道通过 \nolinkurl{writeStream.foreachBatch(_process_batch).trigger(processingTime)} 配置。每个触发周期（默认 5 秒），\texttt{\_process\_batch} 回调被调用：首先将当前微批 DataFrame 中的新事件通过 \texttt{collect()} 收集到 Python 端，追加到内存滑动窗口（\texttt{deque}）中；然后按时间戳清除窗口外的过期事件（默认保留 30 秒窗口）；最后将窗口快照重新构建为 Spark DataFrame，通过 \texttt{groupBy("vehicle\_type").count()} 与 \texttt{agg(F.avg("confidence"))} 等算子执行聚合计算，输出总检测数、唯一轨迹数、平均置信度和车型分布等指标。聚合结果通过回调函数经 Socket.IO 推送至前端监控看板。

该四阶段管道实现了"视频帧 $\rightarrow$ 目标检测 $\rightarrow$ 结构化事件 $\rightarrow$ 文件流 $\rightarrow$ Spark 聚合 $\rightarrow$ 前端展示"的端到端流式处理，在保持 Spark 原生 Structured Streaming API 的规范用法的同时，通过文件源模式避免了引入 Kafka 等外部消息中间件。

\subsection{车牌识别服务与记录入库}

车牌识别能力由 \texttt{plate\_api.py} 与 \texttt{plate\_recognizer.py} 协同提供，服务在 \nolinkurl{/register/recognize} 接口上接收前端上传的图片。接口在进入单（\nolinkurl{.jpg/.jpeg/.png/.bmp/.webp}）、文件大小上限（\texttt{Config.MAX\_UPLOAD\_MB}）与非空判断，校验通过后以"时间戳 + UUID 前缀"拼接文件名落盘至 \texttt{Config.PLATE\_UPLOAD\_DIR}，确保多用户并发上传时文件不互相覆盖。

识别内核 \texttt{PlateRecognizer} 基于 EasyOCR 与 OpenCV 图像预处理构建：对输入图片依次生成双边滤波灰度图、Otsu 二值图、自适应阈值图和 2 倍上采样图等多个预处理变体并送入 OCR 引擎，以提高远距离车牌与光照不均场景下的召回率。OCR 输出通过两级正则模式过滤：严格模式要求首字符为汉字省份简称（如"京"、"粤"）后接 5$\sim$6 位字母数字，宽松模式允许无省份前缀的字符串作为候选，供前端人工补全。所有候选按置信度排序返回，便于用户在多个识别结果中选择最可靠的一条。

无论识别成功、失败还是异常，接口均同步写入 \texttt{plate\_recognition\_record} 记录表：成功写 \texttt{RECOGNIZED}、无有效车牌写 \texttt{FAILED}、捕获异常写 \texttt{FAILED} 并附错误信息。后续 \texttt{/register/vehicle/save} 接口通过 \texttt{recordId} 关联识别记录，在单个数据库事务中依次完成车主查找或新增、车辆档案写入与识别记录状态更新（\texttt{SAVED} 或 \texttt{DUPLICATE}），从而在识别、登记与追溯之间形成可审计的闭环。

\section{RAG 模块实现细节}

RAG 模块由 \texttt{rag\_api.py} 提供接口，底层依赖 \texttt{vehicle\_rag\_service.py}（知识构建与向量检索）、\texttt{embedding\_service.py}（向量编码）、\texttt{milvus\_client.py}（向量库交互）与 \texttt{llm\_service.py}（大模型调用）协同完成。接口包括 \texttt{POST /api/rag/index}（重建向量索引）、\texttt{POST /api/rag/search}（一次性返回）、\texttt{GET /api/rag/stream}（SSE 流式返回）和 \texttt{GET /api/rag/history}（问答历史分页）。流式接口通过 Flask 的 \texttt{Response(generate(), mimetype='text/event-stream')} 实现 SSE 推送，依次发送 \texttt{meta}（候选车辆列表）、\texttt{message}（逐 token 回答）和 \texttt{end}（结束信号）三类事件，问答记录写入 \texttt{rag\_qa\_log} 表。


\subsection{RAG 流程详细说明}

RAG 链路在本系统中的目标不是替代业务规则，而是为管理人员提供可追溯、可解释的辅助问答能力。其执行流程可分为以下阶段：

（1）问题输入与规范化：前端将用户问题提交到 /rag/search 或 /rag/stream，服务端先进行文本清洗、分词、停用词处理和字符标准化，减少噪声对检索结果的干扰。

（2）向量化编码：通过 embedding\_service.py 将问题编码为向量表示，保证语义相近的问题在向量空间具有可比性。

（3）向量库召回：调用 vehicle\_rag\_service.py 与 milvus\_client.py 从向量库中检索 Top-K 候选知识片段，候选来源包括车辆主数据描述、规则文档和历史运维记录。

（4）候选重排与过滤：根据语义相关度与业务字段匹配度对候选片段重排，剔除低相关、过期或与问题主题不一致的片段，降低"检索命中但不可用"的风险。

（5）Prompt 构建：将用户问题、候选上下文、回答约束（如必须给出依据、禁止编造业务字段）组合为结构化提示词，再交由 llm\_service.py 调用模型生成。

（6）结果处理：对模型输出进行格式整理和来源标注，必要时附上关键证据片段，确保结果可复核；随后返回给前端页面进行展示。

（7）日志落库与可追踪：将问题、回答、命中来源、响应时延写入 rag\_qa\_log，用于后续质量评估、故障复盘与提示词优化。

上述七个阶段的串接关系如图\ref{fig:rag_full_process}所示。图中完整呈现了从用户提问输入到回答输出的流程，包括文本规范化、向量化编码、Milvus 向量库 Top-K 召回、候选重排过滤、Prompt 组装、大模型生成与结果处理，以及最终的日志落库与 SSE 流式返回。该流程图用于帮助读者建立 RAG 链路各阶段之间的顺序与依赖关系，说明系统如何通过"检索先行、生成后置"的方式保证回答的可解释性与可追溯性。

\section{系统集成与部署}

\subsection{双后端接口组织与请求转发}

本系统采用"前端统一入口 + 双后端服务"的组织方式。前端开发服务器在 \nolinkurl{vite.config.js} 中配置了两条代理规则：\texttt{/api} 转发至 Java 服务（\texttt{localhost:8080}），\texttt{/py-api} 转发至 Python 服务（\texttt{localhost:5000}），并通过路径重写将 \texttt{/py-api} 统一改写为后端蓝图前缀 \texttt{/api}。该设计使页面层无需关心后端语言边界，接口调用保持同一风格，便于后续替换部署地址或增加网关层。

在请求封装层，前端维护 \texttt{request.js} 与 \texttt{request\_py.js} 两套 Axios 实例，二者都采用统一返回结构（\texttt{code/message/data}）进行拦截处理。Java 侧业务 API 如车辆、车主、停车接口走 \texttt{/api} 通道；Python 侧视频、统计、RAG、车牌识别接口走 \texttt{/py-api} 通道。通过协议统一与路径分离，项目在不引入额外中间件的前提下实现了稳定的跨语言协同。

\subsection{运行部署与启动流程}

系统本地运行按"数据库 $\rightarrow$ Java 服务 $\rightarrow$ Python 服务 $\rightarrow$ 前端"顺序启动。前端通过 Vite 代理接入两个后端；Java 服务提供主业务管理 API；Python 服务提供实时识别、统计推送、RAG 与车牌识别能力。该启动顺序可避免前端初次连接时产生大量错误提示。

Python 服务在启动阶段尝试初始化关键资源，包括 \texttt{plate\_recognition\_record} 与 \texttt{rag\_qa\_log} 表结构、Socket.IO 事件注册、可选模块导入（Spark、YOLO、EasyOCR 等）。其中可选模块导入失败不阻塞服务启动的策略确保了系统在资源不完整环境下仍可运行基础功能。

配置管理方面，项目通过 \texttt{config.py} 和 \texttt{.env} 集中维护参数：视频源路径、帧率与分辨率、YOLO 模型路径与置信阈值、上传目录和大小限制、Milvus 与大模型接口地址等。通过参数化方式，系统在不同设备和数据源条件下可快速迁移。

\subsection{异常处理与降级策略}

系统异常处理遵循"优先保障主流程可用"的原则。对于实时链路，若单帧识别失败或帧发送失败，线程不中断，下一帧继续处理；若视频源读取失败，则通过状态接口反映停止状态并允许重新启动任务。帧发送失败会打印告警但不阻断后续帧处理，避免前端看板完全失效。

在车牌识别模块中，异常路径均写回识别记录表：识别失败写 \texttt{FAILED}，重复入库写 \texttt{DUPLICATE}，成功入库写 \texttt{SAVED}。该设计使异常结果可解释、可追踪。

在 RAG 模块中，系统提供一次性返回接口与流式返回接口双通道。若流式链路受网络影响不稳定，可回退到一次性接口以保障可用性。

\subsection{联调规范与接口契约}

项目联调过程中，问题高发点集中在三类：接口参数契约不一致、统计字段口径不一致、任务结束收尾不一致。针对第一类问题，实践策略是前端必填校验与后端必填校验同步修改；针对第二类问题，统一由后端聚合接口输出总量，前端不再本地重算；针对第三类问题，将手动停止与自然结束统一到同一收尾函数，确保 Structured Streaming 停止与资源释放。

在此基础上，项目形成了可执行规范：（1）新增或变更接口时必须同时检查前端 API 封装、页面校验与后端 Controller 参数；（2）统计字段新增时必须同步检测页和统计页绑定关系；（3）视频链路调整时必须验证 \texttt{start $\rightarrow$ running $\rightarrow$ stop/eof $\rightarrow$ flush} 全状态闭环。

跨模块联调中最常见的非功能缺陷是字段语义漂移。为降低此类风险，系统统一聚合字段契约，前端优先消费 camelCase 字段（如 \texttt{totalCount/currentHourCount/todayCount}），检测服务在 \texttt{video\_api.py} 与 \texttt{monitor\_streaming.py} 中完成模型输出字段到业务指标字段的映射后统一推送，前端不再从检测明细自行推导总量。

前端测试验证交互状态是否与后端实际状态一致，覆盖车辆管理、停车管理、车辆监控、RAG 问答和车辆登记五类页面。以监控页面为例，需同时观察按钮状态（开始/停止识别）、连接状态（聚合流/视频流）、图表状态（趋势与分布）和历史表状态（明细记录）是否一致变化。

在监控页面中，测试重点为 Spark 聚合流与检测流回退的衔接。页面初始化后连接 \texttt{/stream/monitor} 与 \texttt{/stream/monitor-video}，当聚合流短时波动时回退检测流更新曲线。验证标准包括：页面刷新后不出现长时间空值；网络短时波动后可恢复；断开重连后数据可继续追加。

\subsection{Java 业务接口与约束测试}

Java 接口测试采用"参数边界 + 业务约束 + 结果回查"三步法。参数边界验证必填项缺失、格式异常和 ID 不存在时的可读错误信息；业务约束重点验证车牌唯一、在场记录唯一等规则；结果回查则在写入成功后立即读取列表，确认状态字段与关联字段一致。

车辆接口关键用例包括：同车牌重复新增返回失败、修改车牌为已占用值返回失败、设置 \texttt{ownerId} 后车主冗余字段自动更新。停车接口关键用例包括：同一车辆存在 \texttt{PARKING} 记录时再次新增返回失败、车辆离场后可重新入场、分页查询按进入时间倒序返回。

\subsection{Python 实时链路测试}

实时链路测试以任务生命周期完整性为核心，验证 \texttt{start $\rightarrow$ running $\rightarrow$ stop/eof $\rightarrow$ flush} 是否闭环。测试中分别执行手动停止和自然结束两条路径，检查两条路径均能触发统计收尾与资源释放。

对于事件抽取逻辑，重点观察目标短时遮挡后重现、类别短时抖动、多目标交汇时轨迹切换等场景，验证去重后统计不发生明显爆增，以"事件级统计结果可复核"作为准则。

\subsection{RAG 问答链路测试}

RAG 模块测试分为索引构建测试和问答链路测试。索引构建通过调用 \nolinkurl{/api/rag/index} 验证能否从 MySQL 读取车辆文本并完成向量写入。问答链路分别验证 \nolinkurl{/api/rag/search}、\nolinkurl{/api/rag/stream} 与 \nolinkurl{/api/rag/history}，其中流式接口需检查 \texttt{meta/message/end} 三类事件是否完整到达。测试重点关注"是否基于检索结果作答"和"响应过程是否可追踪"。

\subsection{车牌识别模块测试}

车牌识别测试聚焦于"识别准确性可用、流程闭环可达、异常路径可回溯"。具体用例包括：支持格式文件上传、超大文件拦截、非图片文件拦截、识别成功后返回候选列表、人工修正后成功入库、重复车牌入库被拒绝并标记为 \texttt{DUPLICATE}、历史记录可分页查询。该模块强调"人机协同"，即便 OCR 结果不完美，用户仍可通过候选选择与人工修正完成建档。

\section{一致性与稳定性测试}

一致性测试使用同一视频重复回放，对比三处数据：识别明细、事件明细、统计汇总。结果表明：（1）页面展示总量与 Spark Structured Streaming 聚合输出的 \texttt{totalCount} 一致；（2）聚合流与检测流在同一任务周期内统计趋势一致；（3）手动停止与自然结束最终统计一致。

稳定性测试在网络抖动和数据库短时压力场景下进行。系统通过重试与幂等更新保持主流程可用，未出现统计重复写入。连续运行多次任务后资源可正常回收，不存在持续性内存增长。

\section{测试结果量化分析}

系统在三个方面取得提升：（1）可解释性：识别结果与统计结果通过事件层分离，语义分层明确；（2）一致性：监控统计由 Spark Structured Streaming 聚合统一输出，前端不再本地重算，同一指标跨页面一致；（3）可维护性：错误可追踪、接口契约清晰、视频结束收尾流程统一。

\section{集成验证与回归测试}

\subsection{接口日志一致性验证}

通过对 /monitor/start、/monitor/stop、/monitor/status、/api/parking 等接口日志进行关联分析，可观察到一次完整任务从启动到结束的调用链路闭环。

\subsection{业务链路验收}

为验证跨模块数据一致性，本文设计了四条端到端业务链路进行验收。

\textbf{链路一：车辆建档到停车入场。}链路从车辆页面新增档案开始，写入成功后在停车页面选择该车辆并登记入场。验收点包括：车辆列表可查询到新增记录；停车新增时下拉可见该车辆；停车记录状态为 \texttt{PARKING}；再次为同一车辆新增在场记录被拒绝。

\textbf{链路二：视频识别到统计看板。}链路从调用 \nolinkurl{/api/monitor/start} 开始，监控页接收聚合与视频两路流并更新图表。验收点包括：识别框与标签持续刷新；统计卡片与趋势图持续更新；停止或自然结束后不再增长；Structured Streaming 短时不可用时检测流可继续驱动更新。

\textbf{链路三：车牌识别到车辆入库。}链路从上传图片到 \texttt{/register/recognize} 开始，识别成功后用户补全信息并提交 \texttt{/register/vehicle/save}。验收点包括：识别记录表状态更新；车辆主表新增记录可被查询；重复车牌返回冲突并标注原因。

\textbf{链路四：RAG 问答到候选展示。}链路从问答页面提交问题开始，服务端进行向量检索并流式返回答案。验收点包括：前端先收到 \texttt{meta} 候选列表，再收到文本 token；结束事件触发后加载状态关闭；同类问题重复提问可获得结构稳定的回答。

\subsection{跨模块回归测试}

回归测试覆盖"车辆管理—停车管理—实时识别—统计查询—RAG 问答"全链路。修复类别映射和统计口径后，监控页面车型显示准确，聚合流与检测流回退机制协同正常，页面各处统计展示一致。

\subsection{回归测试执行策略}

为避免修复引入退化，回归测试采用固定顺序：先接口、后页面、再链路。接口层确保单点正确；页面层确保交互不退化；链路层确保跨模块语义一致。每次修复后至少回放一次同源视频并复核统计结果。

回归过程中保持两项工程纪律：（1）统计口径变更必须同步更新展示绑定；（2）新增参数必须同步更新前端校验和后端校验。

\section{误差来源与案例复盘}

\subsection{误差传递链路分析}

识别侧误差主要来自光照剧烈变化、遮挡重叠和目标远近变化；统计侧误差主要来自事件边界判定与窗口切换时机；展示侧误差主要来自刷新周期与网络延迟。三类误差沿"识别—事件—统计—展示"链路逐级传递，需要系统性治理。

项目采用的治理方法包括：识别侧通过阈值和时序平滑抑制抖动；事件侧通过去重键和方向约束减少重复；统计侧通过幂等更新保障一致；展示侧通过直连聚合接口避免二次计算。经该组合治理后，系统在多轮样本上的一致性表现更为稳定。

为验证可复现性，本文在同一输入样本上进行多次回放测试并对比最终统计结果。实验表明，在相同配置下系统输出具备稳定性；当配置变化时，结果变化方向与预期一致，说明系统具备可控可解释的工程特性。此外，本文采用"失败样本复盘"方法：对统计偏差样本保留原始输入、识别结果、事件明细与汇总结果，按时间线回溯定位偏差来源，将模糊问题转化为可定位问题。

\subsection{典型案例复盘}

\textbf{案例 A：货车被显示为 car。}现象：监控页面所有目标均显示 car。定位：前端类别映射未使用后端 class\_id。修复：按类别 ID 映射显示 car/bus/truck，并增加默认兜底。验证：回放同视频，货车目标正确显示 truck。

\textbf{案例 B：统计总量偏少。}现象：监控聚合总量小于识别页面累计数量。定位：前端仅消费聚合流，Structured Streaming 短时不可用时无回退。修复：统一事件模型并增加聚合优先、检测流回退机制。验证：页面在聚合可用与不可用场景下均连续更新。

\textbf{案例 C：停车新增返回 400。}现象：提交新增记录后返回 400，且选项列表缺失。定位：前端时间格式与后端校验规则不一致，下拉选项来源错误。修复：统一时间格式、改为后端动态拉取选项、补充字段校验提示。验证：同数据重复提交可稳定成功，错误提示可定位字段。


\section{本章小结}

本章通过功能测试、一致性测试、稳定性测试与链路级验收，验证了系统在真实业务场景中的可用性与可靠性。测试结果表明，统计一致性、接口稳定性与收尾完整性均有显著改善。误差来源分析与案例复盘进一步提供了可定位、可复现的诊断方法。

\chapter{总结与展望}

\section{研究总结}

本文围绕智能小区车辆管理系统，完成了前后端协同、视频识别、车辆监控聚合与 RAG 辅助问答的整体设计与实现。

在系统架构层面，采用 Vue 3 + Spring Boot + Flask 的前后端分离与双后端协同架构，通过 Vite 代理实现跨语言服务的透明接入，使业务管理与 AI 服务在统一接口规范下协同工作。MySQL 承担业务数据持久化，Milvus 承担向量检索，二者分工明确，避免了单一存储引擎在异构查询场景下的性能瓶颈。

在技术方案层面，针对项目迭代中暴露的三类核心问题，分别完成了针对性改进：（1）修复车辆类型映射，解决实时监控页面中所有目标统一显示为 car 的问题，使 car、truck、bus 三种车型可正确区分；（2）统一统计口径并引入 Spark Structured Streaming 聚合，解决识别数量与监控统计数量不一致的问题，使统计一致性明显提升；（3）统一表单与接口校验规则，解决停车新增接口 400 错误频发与下拉选项缺失问题，使接口稳定性明显改善。

在工程实践层面，项目沉淀了可执行的联调规范：前后端参数校验同步修改、统计字段新增同步绑定展示、视频链路调整同步验证全状态闭环。通过接口级、链路级和回归级三层测试体系，系统功能正确性与跨模块一致性均得到验证。

\section{不足分析}

当前系统在以下方面仍有改进空间：

（1）视觉识别鲁棒性不足。在极端光照（强逆光、夜间无灯）和严重遮挡（车辆紧密排列、树荫遮蔽）场景下，YOLOv8n 的检测稳定性下降，导致事件抽取层输入质量降低。当前系统仅依赖置信度阈值和时序平滑进行后处理，缺少针对场景自适应的增强策略。

（2）单摄像头覆盖范围受限。系统当前仅支持单路视频源输入，无法实现多摄像头跨区域事件关联与全域车流分析。对于大型社区的多出入口场景，需要引入跨摄像头 ReID（重识别）技术才能构建完整的车辆轨迹。

（3）RAG 模块能力边界有限。当前 RAG 以结构化车辆数据为主要知识源，对非结构化运维文档和历史工单的覆盖不足。问答结果的结构化程度和可视化解释能力有待增强，尚不具备自动归因和趋势分析功能。

（4）权限控制未完整实现。Java 后端 Spring Security 当前配置为 \texttt{permitAll} 开放策略，仅用于联调阶段。在生产部署前需补充基于角色的访问控制（RBAC），以保障数据安全性。

\section{未来工作}

后续将从以下方向继续优化：

（1）提升复杂场景识别能力。计划引入更稳定的多目标跟踪算法（如 ByteTrack 或 BoT-SORT）替换当前基于 IoU 的简单跟踪器，并探索场景自适应参数调节机制，根据光照条件和目标密度动态调整检测阈值与帧间隔，以提升夜间和拥堵场景下的识别效果。

（2）增强多摄像头融合能力。计划引入跨摄像头 ReID 技术，通过外观特征与时空约束实现跨区域车辆匹配，构建全域车流分析能力。同时扩展 Spark 聚合模块以支持多源事件流的统一窗口聚合。

（3）深化 RAG 智能问答能力。计划扩展知识源覆盖范围，纳入运维日志、告警记录和历史工单；引入问答结果的结构化展示（如表格化统计、趋势图生成）；探索自动归因链路，使系统能够在检测到统计异常时主动推送可能的原因分析。

（4）完善系统安全与部署方案。计划在 Spring Security 中实现完整的 RBAC 权限控制，并引入 JWT 令牌机制替代当前的 localStorage 登录态管理。在部署层面，探索容器化方案（Docker Compose）以简化多服务编排和环境一致性管理。
